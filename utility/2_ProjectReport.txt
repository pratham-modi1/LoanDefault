Stage 4: Strategic Decision — SMOTE vs Threshold Optimization
Objective

At this stage, the primary objective is to determine the most effective next step for improving model performance. Two potential approaches were considered:

Applying SMOTE (Synthetic Minority Oversampling Technique) to address class imbalance
Performing Threshold Optimization to improve decision-making

This section provides a structured comparison and justifies the chosen direction.

Current Model Behavior: Problem Diagnosis

From Stage 3 results:

Recall ≈ 0.73 → The model successfully identifies most defaulters
Precision ≈ 0.22 → High number of false positives

This leads to a critical conclusion:

The model has already learned the underlying patterns in the data effectively, and further improvements are no longer limited by model learning capacity.

Instead, the bottleneck has shifted to:

Decision Strategy (i.e., how predictions are converted into final classifications)

Understanding SMOTE

SMOTE (Synthetic Minority Oversampling Technique) is a data balancing method used to address class imbalance.

How SMOTE Works
It generates synthetic samples for the minority class (defaulters)
Instead of duplicating data, it creates new points by interpolating between existing minority samples
This artificially increases the representation of defaulters in the training dataset
Purpose of SMOTE

SMOTE is typically useful when:

The model fails to learn minority class patterns
Recall is low (model misses defaulters)
The dataset is highly imbalanced and the model is biased toward the majority class
Limitations of SMOTE in This Context

In the current project:

Recall is already high (~73%)
The model is successfully identifying defaulters
PR-AUC has plateaued across models

This indicates that:

The model is not struggling to learn minority patterns

Applying SMOTE at this stage may:

Introduce synthetic noise into the dataset
Increase overfitting risk
Potentially worsen precision by making the model overly aggressive
Understanding Threshold Optimization

Machine learning models output probabilities, not final decisions.

By default, predictions are converted using a threshold of 0.5:

If probability > 0.5 → Default
Else → Non-default

However, in imbalanced datasets (~6.6% default rate), this threshold is not optimal.

What Threshold Optimization Does
Adjusts the probability cutoff used for classification
Controls the trade-off between Precision and Recall
Allows alignment with business objectives
Why It Matters

At the current stage:

The model ranks borrowers effectively (high PR-AUC)
But makes suboptimal classification decisions (low precision)

This means:

The model knows who is risky, but is not deciding how risky is risky enough

Methodology

Instead of using the default classification threshold of 0.5, which is not suitable for highly imbalanced datasets (~6.6% defaulters), threshold optimization was performed.

Phase 1: Feature Engineering (Breaking the Ceiling)
The Problem:
Tree-based algorithms are literal. When fed 30-day, 60-day, and 90-day late payment counts as separate variables, the model struggles to inherently understand the compounding severity of chronic delinquency.

The Solution:
Domain-specific financial features were engineered to provide "smarter" signals to the model, translating raw data into direct indicators of financial stress.

Total_Late_Events: A simple sum of all late payment columns to represent the raw volume of delinquency.

Weighted_Late_Score: Applied a severity multiplier (30-day*1 + 60-day*2 + 90-day*3). This mathematically forced the model to penalize severe 90-day delinquencies exponentially harder than minor 30-day delays.

Credit_Utilization_Warning: A binary flag (0 or 1) triggered if a user's credit utilization exceeded 85%, creating a sharp decision boundary for extreme debt.

Income_Per_Dependent: Calculated by dividing Monthly Income by (NumberOfDependents + 1). This normalized income to reflect actual purchasing power and household financial strain.

The Result:
Applying these features immediately broke the PR-AUC ceiling. The XGBoost baseline PR-AUC rose from 0.364 to 0.368, proving the model found genuine new predictive value in the engineered columns.

The Solution:

## 1. Introduction

In this project, **SHAP (Shapley Additive Explanations)** was used to interpret the predictions of a machine learning model for loan default risk.

Traditional ML models act as **black boxes**, providing predictions without explanations. Since loan approval is a **high-stakes financial decision**, interpretability becomes critical. SHAP was applied to:

* Understand **why a prediction was made**
* Identify **key drivers of default risk**
* Validate that the model is learning **meaningful patterns**

---

## 2. What is SHAP?

SHAP is based on **Shapley values** from game theory.

* Each feature is treated as a *player*
* The prediction is the *final outcome*
* SHAP distributes the contribution of each feature **fairly**

### Core Idea:

> Final Prediction = Base Value + Sum of Feature Contributions

---

## 3. SHAP Workflow Implemented

The following steps were used:

1. Extract trained model (XGBoost from ensemble)
2. Initialize SHAP explainer
3. Compute SHAP values on sample data
4. Analyze:

   * Individual prediction (local explanation)
   * Entire dataset (global explanation)

---

## 4. Key Components Explained

### 4.1 Explainer

The `Explainer` object learns:

* Model behavior
* Feature relationships

It acts as a **bridge between model and interpretation**

---

### 4.2 SHAP Values

For each prediction:

* Each feature gets a **numerical contribution**
* Positive → increases risk
* Negative → decreases risk

---

### 4.3 Base Value

* Represents **average model prediction**
* Starting point before feature contributions

---

### 4.4 Final Prediction

* Computed as:

> Base Value + SHAP Contributions

For interpretability, log-odds were converted to **probability**

4. The XGBoost Overhaul & Automation
Instead of manually guessing the best scale_pos_weight to handle the 93/7 class imbalance, we built an automated scanner.

🟡 MODIFIED (XGBoost Hyperparameters):

scale_pos_weight: Changed from a static 18 to a dynamic variable.

eval_metric: Changed from the default (logloss) to 'aucpr'. This was massive—we told the model to explicitly train for the metric we actually care about.

learning_rate: Dropped from 0.02 to 0.01 (slower learning).

n_estimators: Dropped from 500 to 400.

max_depth: Dropped from 7 to 5 (simpler, less prone to overfitting).

min_child_weight: Lowered from 40 to 20.

🟢 ADDED (The Scan Loop):

Added a for spw in [3, 5, 6, 7, 10]: loop.

The script now trains 5 miniature XGBoost models in a row, tests their PR-AUC, automatically memorizes which weight performed best (in your case, it was 3), and then builds the final best_xgb using that winning number.
