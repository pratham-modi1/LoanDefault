STAGE 2: EXPLORATORY DATA ANALYSIS (EDA)

Part 1: The Strategic Purpose of EDA

Why EDA Is Non-Negotiable

Exploratory Data Analysis (EDA) is a mandatory foundation step in any serious machine learning workflow. If raw data is fed directly into an algorithm without first understanding its structure, patterns, weaknesses, and limitations, the model will not truly learn the business problem. Instead, it will often memorize the flaws present in the dataset.

In that sense, EDA serves as an X-ray before surgery. It reveals hidden landmines, identifies which features contain genuine predictive signal, and highlights which features are only introducing noise. This makes it possible to approach model building with clarity rather than guesswork.

What EDA Helps Us Achieve

EDA provides several important advantages:

Avoiding False Victories

EDA protects us from falling into statistical traps, such as building a model that appears highly accurate on paper but fails in real-world usage. A model can look impressive in a report while still being practically useless if it has learned the wrong patterns.

Filtering the Noise

EDA saves significant time later in the pipeline by identifying which features should be transformed, which should be retained, and which should be dropped entirely. This reduces debugging effort and helps prevent wasted experimentation.

Establishing Ground Truth

EDA gives us a mathematical baseline. Later, when the AI model generates predictions, we can compare its behavior against our exploratory findings to verify that the model is learning meaningful financial behavior rather than hallucinating relationships that do not actually exist.

Part 2: The Analytical Execution

Target Analysis and Class Imbalance

What We Did

We first analyzed the distribution of the target variable, SeriousDlqin2yrs, which indicates whether a borrower defaulted.

The Findings

The dataset shows severe class imbalance. Approximately 93.37% of borrowers are non-defaulters, while only 6.63% are defaulters.

Interpretation: The Accuracy Trap

Because loan default prediction is a highly imbalanced problem, a model trained on raw data can easily learn to always predict “no default” and still achieve about 93% accuracy. Although this may look good mathematically, it is useless from a business standpoint because it fails to identify the minority class that actually matters.

How This Improves the Model Going Forward

For this project, standard accuracy is formally rejected as a meaningful primary metric. Instead, the model must be evaluated strictly using Recall, because catching actual defaulters is the highest priority, and Precision, because we also want to avoid falsely denying good customers.

This analysis also makes it clear that class-balancing methods must be used during training, such as SMOTE or algorithmic class weighting, so that the model gives proper importance to the 6.63% minority class.

Primary Evaluation Metric Definition: Audit Update

Because of the extreme 93.37% versus 6.63% imbalance, standard machine learning metrics such as Accuracy and ROC-AUC are formally rejected. A lazy algorithm could still achieve a high ROC-AUC by simply predicting “No Default” for everyone, while completely ignoring the minority class.

Moving into model training, the models will therefore be evaluated strictly using Precision-Recall Area Under Curve (PR-AUC) and its underlying components. The hierarchy of importance is as follows:

Primary: Recall — we must maximize the capture rate of actual defaulters, because missing a defaulter is the highest cost to the business.

Secondary: Precision — we must monitor false positives so that safe borrowers are not aggressively rejected.

Tertiary: F1-Score — this will measure the overall harmonic balance between Precision and Recall.

Correlation Matrix: The Strategic Filter

What We Did

Before plotting anything, we generated a mathematical correlation matrix to test how strongly every single feature column correlates with defaulting.

Why We Did It First

The Golden Rule of Business Data Science is: calculate everything, but only plot what matters. If a dataset has thousands of columns, creating graphs for every single one would lead to “Death by a Thousand Plots” and waste stakeholder time. The correlation matrix acts as a filter, showing us which features actually drive the model so we know where to focus our attention.

The Findings

High Correlation, Strong Signals

Late payment history, especially NumberOfTimes90DaysLate, and Credit Utilization emerged as the dominant warning signs. Their correlations were in the range of 0.24 to 0.34, which makes them meaningful predictors.

Low Correlation, Weak Signals

NumberOfDependents, MonthlyIncome, and DebtRatio showed near-zero linear correlation, below 0.05. Features such as NumberOfOpenCreditLines showed a correlation of -0.03, which is practically zero.

How This Improves the Model Going Forward

This step directly sets up the answer key for the final model evaluation and gives us a concrete troubleshooting plan.

Establishing the SHAP Answer Key

Advanced algorithms like Random Forest are notoriously complex black boxes, which makes it difficult to explain why they approved or denied a loan. SHAP (SHapley Additive exPlanations) will be used to open up this black box and force the AI to rank exactly which features influenced its decisions.

Because of the EDA, we are not approaching this blindly. We have already mathematically proven that 90-days-late behavior is the strongest real-world signal. Therefore, when SHAP is run later, we expect the model to strongly agree and assign the highest predictive weight to that feature.

The Audit Mechanism: Spotting Hallucinations

If the final model’s SHAP analysis claims that a feature such as MonthlyIncome or NumberOfDependents is the primary reason for a default, we will immediately know that the model is flawed. That would suggest the algorithm is hallucinating by memorizing noise instead of learning genuine financial behavior, and the model would need to be audited.

A Pre-Planned Defense Against Overfitting

By identifying weak signals now, such as DebtRatio and MonthlyIncome, which showed near-zero correlation, we have created a proactive defense against overfitting. If the Stage 3 model begins to overfit, meaning it performs perfectly on training data but poorly on new data, we will not need to guess how to fix it. We will already have a pre-planned dimensionality reduction strategy: remove these weak and noisy features so the model can focus entirely on the strongest signals.

Multicollinearity Assessment: The Echo Chamber Check

What We Did

We examined feature-to-feature correlation to see how input variables relate to each other. In simple terms, we asked whether two variables are repeatedly telling the same story. For example, is someone who is 30 days late also automatically 90 days late?

Interpretation: The Courtroom Analogy

Multicollinearity happens when two features communicate essentially the same information. It is like two witnesses in a courtroom shouting the exact same testimony at the same time. This creates confusion. In machine learning, highly correlated features can confuse tree-based models, distort feature importance, and make truly important variables appear artificially weak. It can also completely damage SHAP analysis.

The Findings

Our late payment features overlap only by about 25% to 30%.

How This Improves the Model Going Forward

This is a strong and successful outcome. It proves that the late payment features represent distinct stages of financial distress rather than duplicates of the same signal. As a result, we can safely retain all of them without confusing the model.

Univariate and Bivariate Analysis: Visualizing the Signals

What We Did

We took the important and tricky features identified by the correlation matrix and plotted them. For discrete count variables such as late payment counts, we used bar charts. For continuous variables such as Age and Income, we used density curves, which act like overlapping mountains. This allowed us to compare the visual shape of Good Borrowers versus Defaulters.

Interpretation: The Golden Rule of Density Plots

High correlation means high separation. If two groups are visually separated, the model can more easily distinguish them.

Zero correlation means perfect overlap. If two groups look identical, the feature is not useful on its own.

The Findings

Absolute Default Triggers

The bar charts showed that while 30-day delays are relatively common, crossing the 90-day threshold is the definitive point of no return for borrowers. This makes the 90-day late-payment features extremely important.

Age Separability

The density curves showed a clear shift in borrower age. Defaulters are more concentrated in the 30 to 40 age range, while safe borrowers peak in the 50 to 60 age range. This gives the model a clear dividing line.

Income and Debt Overlap

The curves for MonthlyIncome and DebtRatio almost perfectly overlap. Visually, this proves that high income alone or high debt alone does not determine default risk.

Why We Skipped Plotting Other Columns

We did not plot columns such as NumberRealEstateLoansOrLines because the correlation matrix already showed that their relationship with the target was zero. Plotting them would only produce identical, perfectly overlapping mountains and would not teach us anything new. It would also clutter the report unnecessarily.

How This Improves the Model Going Forward

We now have both visual and mathematical proof of how the model is likely to separate the data. Based on these findings, we expect the tree nodes to split heavily on Age and 90-days-late behavior, while treating Income as a secondary or tertiary factor.

Validating Primary Signals: Credit Utilization

We performed a deeper analysis of RevolvingUtilizationOfUnsecuredLines, which is one of the strongest predictive signals in the dataset, with a correlation of 0.24.

Statistical grouping revealed a major behavioral gap. Non-defaulters use an average of 29.3% of their available credit, while defaulters operate at a highly stressed 68.5%. Density plots also confirmed this difference visually, with defaulters peaking heavily near 100% utilization.

This proves that RevolvingUtilizationOfUnsecuredLines is a primary and highly valid splitting feature for tree-based models, and it will be a key focus in the final model evaluation.

STAGE 2.5: DATA SPLITTING and RAW BASELINE PREPARATION

Before building models or engineering new features, we prepared the raw data so that we could establish a clean and unbiased performance baseline.

Train/Validation Split: The Mock Exam Strategy

What We Did

We split the primary training dataset into an 80% Training Set and a 20% Validation Set, using strict stratified sampling to preserve the 93% to 6% class imbalance.

Why We Did It: The Data Leakage Threat

Even though a separate test.csv file exists, that file acts as the Final Exam. In real-world projects or Kaggle competitions, the target values in the test set are hidden. If we repeatedly tune the model based on performance on the final test set, the model is no longer learning—it is merely memorizing the test data.

The Validation Set serves as our Mock Exam. It gives us a safe, private environment to tune hyperparameters and fix mistakes without cheating or causing data leakage.

Feature Scaling: The Level Playing Field

What We Did

We applied StandardScaler to normalize the mathematical scale of the features.

Strategic Reasoning: Model Agnosticism

Our primary algorithm for this project is Random Forest. Since Random Forest is a tree-based model that splits data using thresholds, it is mathematically immune to unscaled data. However, a strong data science pipeline should not rely on just one algorithm.

By scaling the data, we create a level playing field for distance-based algorithms such as Logistic Regression or KNN. Without scaling, a feature like MonthlyIncome, which may be around 50,000, would mathematically dominate a feature like DebtRatio, which may be around 0.35, causing distance-based models to perform badly.

Execution: Preventing Leakage

To maintain scientific validity, the scaler was fitted only on the 80% training data and then applied to the validation data. This ensures that no information from the validation set leaked into the training environment.

A Tiny Example to Understand fit and transform

Let us consider a very small dataset with only one column: Age.

Train.csv contains four values: 20, 30, 40, 50

Test.csv contains two values: 60, 70

The scaling formula is always: (Value - Mean) / Standard Deviation.

For easy understanding, assume the training standard deviation is exactly 10.

What fit(Train) exactly does

When scaler.fit(X_train) is executed, the code looks at the four training values and calculates the mean.

Train Average = (20 + 30 + 40 + 50) / 4 = 35

Train Standard Deviation = 10

At this stage, the data itself does not change. The values remain 20, 30, 40, and 50.

What fit actually does is store the numbers 35 and 10 in memory. It simply learns the formula variables.

What transform(Train) exactly does

When scaler.transform(X_train) is called, the stored values 35 and 10 are used to transform the data.

Row 1 (20): (20 - 35) / 10 = -1.5

Row 2 (30): (30 - 35) / 10 = -0.5

Row 3 (40): (40 - 35) / 10 = 0.5

Row 4 (50): (50 - 35) / 10 = 1.5

Now the training data is centered around 0 and scaled into a smaller numeric range.

The Correct Way: transform(Test)

Now consider the test data: 60 and 70.

When scaler.transform(X_test) is used, it does not calculate a new average. Instead, it reuses the 35 and 10 learned from the training data and applies the same formula to the test data.

Row 1 (60): (60 - 35) / 10 = 2.5

Row 2 (70): (70 - 35) / 10 = 3.5

This is correct because the model now sees that the test values are much larger than the training values.

The Mistake: fit(Test)

If we accidentally fit the test data, the code would calculate a new average using only the test values.

Test Average = (60 + 70) / 2 = 65

Test Standard Deviation = 5

Then it would transform the test values using this new formula.

Row 1 (60): (60 - 65) / 5 = -1.0

This creates a serious problem. The 60-year-old in the test set now gets a score of -1.0, while earlier a 20-year-old in the training set got -1.5.

Because the scale was recalculated separately, the meaning of the number has changed. The model would incorrectly interpret the 60-year-old as being similar to a young 20-year-old. That would completely ruin the predictions.

This is why we only fit on the training data. We build the measuring stick once and then use the exact same measuring stick on the test data.

Since we have now officially completed Scaling (Step 9) with the Stage 2.5 code, there are only two things left from that original checklist:

Step 5: Feature Binning, such as converting “Age 34” into a “Young Adult” category.

Step 8: Feature Engineering, such as mathematically combining all late-payment columns into one Total Late Score.

However, this is intentional. We decided to pause those steps because we need to build a raw baseline model first. If we engineer several advanced features immediately and the model gets an 80% score, we would have no way to know whether the new features actually improved performance or secretly made it worse.

ISSUE 2: Unvalidated Feature Engineering (MonthlyIncome_was_missing)

The Problem

In Stage 1, a new feature named MonthlyIncome_was_missing was created. This column equals 0 when income was reported and 1 when income was left blank. However, the auditor flagged this feature because we never formally proved that it works.

If people who leave their income blank default at the exact same rate as people who report their income, then this feature is just random noise. Feeding noise into a tree model can cause overfitting. Therefore, we must prove to the auditor that this column contains real mathematical signal.

Validating Engineered Signals: Missing Income Data

We validated the MonthlyIncome_was_missing indicator to make sure it provides true predictive signal rather than random noise.

The analysis showed that borrowers who reported income default at a rate of 6.89%, while borrowers with missing income data default at a distinctly lower rate of 5.50%.

Because there is a clear statistical divergence between the two groups, this engineered feature provides valid predictive value and will be retained for modeling.

Stage 4: Strategic Decision — SMOTE vs Threshold Optimization
Objective

At this stage, the primary objective is to determine the most effective next step for improving model performance. Two potential approaches were considered:

Applying SMOTE (Synthetic Minority Oversampling Technique) to address class imbalance
Performing Threshold Optimization to improve decision-making

This section provides a structured comparison and justifies the chosen direction.

Current Model Behavior: Problem Diagnosis

From Stage 3 results:

Recall ≈ 0.73 → The model successfully identifies most defaulters
Precision ≈ 0.22 → High number of false positives
PR-AUC ≈ 0.364 (consistent across all models)
Key Insight

All models (Logistic Regression, Random Forest, XGBoost) achieved nearly identical PR-AUC scores, despite increasing complexity.

This leads to a critical conclusion:

The model has already learned the underlying patterns in the data effectively, and further improvements are no longer limited by model learning capacity.

Instead, the bottleneck has shifted to:

Decision Strategy (i.e., how predictions are converted into final classifications)

Understanding SMOTE

SMOTE (Synthetic Minority Oversampling Technique) is a data balancing method used to address class imbalance.

How SMOTE Works
It generates synthetic samples for the minority class (defaulters)
Instead of duplicating data, it creates new points by interpolating between existing minority samples
This artificially increases the representation of defaulters in the training dataset
Purpose of SMOTE

SMOTE is typically useful when:

The model fails to learn minority class patterns
Recall is low (model misses defaulters)
The dataset is highly imbalanced and the model is biased toward the majority class
Limitations of SMOTE in This Context

In the current project:

Recall is already high (~73%)
The model is successfully identifying defaulters
PR-AUC has plateaued across models

This indicates that:

The model is not struggling to learn minority patterns

Applying SMOTE at this stage may:

Introduce synthetic noise into the dataset
Increase overfitting risk
Potentially worsen precision by making the model overly aggressive
Understanding Threshold Optimization

Machine learning models output probabilities, not final decisions.

By default, predictions are converted using a threshold of 0.5:

If probability > 0.5 → Default
Else → Non-default

However, in imbalanced datasets (~6.6% default rate), this threshold is not optimal.

What Threshold Optimization Does
Adjusts the probability cutoff used for classification
Controls the trade-off between Precision and Recall
Allows alignment with business objectives
Why It Matters

At the current stage:

The model ranks borrowers effectively (high PR-AUC)
But makes suboptimal classification decisions (low precision)

This means:

The model knows who is risky, but is not deciding how risky is risky enough

Comparison: SMOTE vs Threshold Optimization
Aspect	SMOTE	Threshold Optimization
Purpose	Improve learning of minority class	Improve decision boundary
Use Case	Low recall / weak model learning	High recall, low precision
Data Impact	Modifies dataset with synthetic samples	No change to data
Risk	Overfitting, noise introduction	Minimal risk
Relevance to Current Problem	Low	High
Final Decision

Based on the analysis:

Threshold Optimization is selected as the immediate next step

Justification
The model has already achieved strong recall and stable PR-AUC
Performance plateau indicates learning is no longer the bottleneck
The primary issue is excessive false positives (low precision)
Threshold tuning directly addresses this precision–recall imbalance


Stage 5: Threshold Optimization and Final Model Selection
Objective

After observing that all models achieved nearly identical PR-AUC (~0.364), it was concluded that model performance had plateaued in terms of learning capability. The remaining limitation was due to suboptimal classification decisions rather than the model’s ability to distinguish between classes.

The objective of this stage was to improve model performance by optimizing the decision threshold instead of modifying model architecture or applying resampling techniques.

Methodology

Instead of using the default classification threshold of 0.5, which is not suitable for highly imbalanced datasets (~6.6% defaulters), threshold optimization was performed.

The process involved:

Extracting predicted probabilities using predict_proba()
Computing Precision and Recall across all thresholds using the Precision-Recall curve
Calculating F1 Score at each threshold
Selecting the threshold that maximizes the F1 Score

This approach ensures a mathematically optimal balance between Precision and Recall.

Contrary to typical expectations for imbalanced datasets, the optimal threshold was significantly higher than 0.5.

This indicates that:

The model assigns high probability scores only to strongly confident defaulters, requiring a stricter decision boundary.
